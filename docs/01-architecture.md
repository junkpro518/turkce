# 01 — المعمارية

## المبدأ المركزي
**نفصل "الدماغ التربوي" عن "سباكة العميل".** المحرّك التربوي مستقل عن الواجهة (interface-agnostic)، والعملاء (تيليجرام، ويب) **رفيعون** يتكلّمون معه. وكل شيء يتفرّع من **نموذج المتعلّم**.

> العميل لا يحوي منطقاً تربوياً — هذا بالضبط ما قتل `TR_AR_AI` (وضعت المنطق في جسر رفيع ثم حذفته للاتكاء على Hermes).

## الطوبولوجيا

```
        ┌─────────── المحرّك التربوي (lib/ — TypeScript) ───────────┐
        │  1. نموذج المتعلّم   2. المنهج المتكيّف   3. المنسّق الذكي  │
        │  4. محلِّل التصحيح   5. SRS (FSRS)        6. التقدّم        │
        │            ↕ OpenRouter (المعلّم + المحلّل)                │
        └───────────────┬───────────────────────┬──────────────────┘
                        │                       │
              عميل أساسي │                       │ عميل خفيف
          ┌─────────────┴────┐            ┌──────┴──────────┐
          │ بوت تيليجرام      │            │ ويب Next.js خفيف │
          │ (grammY/webhook) │            │ (لوحة + مراجعة)  │
          └──────────────────┘            └─────────────────┘
                        │
                  Supabase (الحالة الدائمة)
```

**كله TypeScript واحد**: تطبيق Next.js يستضيف المحرّك (`lib/`) + webhook التيليجرام (API route) + صفحات الويب الخفيفة. لا انقسام Python/TS (الذي عانت منه النسختان).

## الوحدات الست للمحرّك

كل وحدة **مركّزة، ذات واجهة واضحة، قابلة للاختبار وحدها**.

1. **🧠 نموذج المتعلّم (Learner Model) — العمود الفقري**
   تمثيل دائم عنك: CEFR لكل مهارة · الأهداف (متغيّرة، ذات أولوية) · خريطة إتقان القواعد · معرفة المفردات (حالة SRS) · أنماط الأخطاء. يُقرأ قبل كل دور، يُحدَّث بعده. التفاصيل في [03](03-pedagogy.md).

2. **🧭 محرّك المنهج المتكيّف (Curriculum Engine)**
   شجرة مهارات CEFR (في الكود) + دالة تختار "التركيز التالي" من نموذج المتعلّم. تُعيد الحساب كلما تغيّر الإتقان أو الهدف. **دالة نقية مُختبَرة.**

3. **💬 المنسّق + المعلّم الذكي (Orchestrator)**
   لكل دور: يبني سياقاً متدرّجاً ← ينادي المعلّم (LLM) الذي **يقرّر نمط التفاعل بحُكمه**. كود صريح، لا graph. التفاصيل في [04](04-conversation-flow.md).

4. **🔍 محلِّل التصحيح العميق (Analyzer)**
   نداء AI منفصل ومنظّم (Zod) يحلّل رسالة الطالب بتصنيف نحوي تركي حقيقي ويشرح *السبب*. يعمل في الخلفية، لا يحجب الرد.

5. **🔁 وحدة SRS**
   مجدول **FSRS** حتمي. يلتقط المفردات من المحادثة، يجدول المراجعة. **دوال نقية مُختبَرة.**

6. **📊 التقدّم**
   اشتقاق من نموذج المتعلّم: CEFR لكل مهارة، هدف المنهج، أبرز الضعف، مستحقات SRS، streak/XP.

## التقنيات

| الطبقة | الاختيار | لماذا |
|--------|----------|-------|
| الإطار | **Next.js 15** (App Router) | يستضيف المحرّك + webhook + الويب في كود واحد |
| اللغة | **TypeScript** | أمان أنواع عبر كل الطبقات |
| قاعدة البيانات | **Supabase (PostgreSQL)** | حالة دائمة موثوقة |
| ORM | **Drizzle** | أنواع + migrations نظيفة (تحسين عن client الخام) |
| طبقة الـ AI | **Vercel AI SDK** | streaming + مخرجات منظّمة (`generateObject` + Zod) فوق OpenRouter |
| تحقّق المخرجات | **Zod** | schemas صارمة لكل نداء AI تحليلي |
| التيليجرام | **grammY** | مكتبة TS حديثة، webhook، أزرار |
| الاختبار | **Vitest** | اختبار الدوال النقية التربوية |
| مزوّد النماذج | **OpenRouter** | مرونة النماذج، نداء مباشر (لا وسيط هشّ) |

## ما لا نستخدمه (وقرارات صريحة)

| مرفوض | السبب |
|--------|-------|
| **LangGraph / LangChain** | بيئتها Python؛ نسخة JS ثقيلة التجريد؛ LangGraph مُصمّم لوكلاء معقّدين ننحرف معهم. `TR_AR_AI` جرّبها وفشلت. |
| **Hermes Agent** | عدم تطابق معماري (وكيل عام لا تطبيق تربوي)؛ يفتقر لنواتنا (SRS/منهج/تتبّع)؛ هشاشة 502 مُثبتة في النسختين. |
| **معماريات وكيلية تتحكّم بالنظام كله** | تُسلّم الوكيل المنهج والجدولة فينحرف ويعيد السطحية. الوكيل يقود **المحادثة** فقط. |

## النشر
عملية Node دائمة على VPS (Docker) — أنسب لـ webhook + التحليل الخلفي + التيليجرام من serverless edge. تفاصيل التشغيل تُحسم في خطة التنفيذ.
