# 05 — تصميم بوت التيليجرام

> الواجهة الأساسية. البوت **عميل رفيع** يتكلّم مع محرّكنا — لا جسر يفوّض الذكاء لإطار خارجي (خطأ `TR_AR_AI` القاتل).

## التقنية
- **grammY** (TypeScript) + **webhook** (لا long-polling) — استجابة أسرع، يندمج مع Next.js API route.
- يدخل لنفس محرّك `lib/` الذي يخدم الويب.

## تجربة المحادثة
- **"streaming" = تحرير الرسالة تدريجياً** كل ~1.5ث + مؤشر `typing`. يبدو حيّاً (التيليجرام لا يدعم token-by-token).
- **تقسيم الردود > 4096 حرف** إلى رسائل متتابعة (إصلاح الفشل الصامت في النسخة القديمة).
- **اختبارات** عبر أزرار inline (callback): اختيار ← تلوين الصح/الخطأ ← شرح.
- **مراجعة SRS** عبر أزرار: عرض الكلمة ← كشف ← تقييم (نسيت/صعب/جيد/سهل).
- **تنبيهات يومية** أصلية (مهمة مجدولة) — تذكير بالمراجعة أو جلسة.

## الحالة والموثوقية (إصلاح أخطاء النسخة الفاشلة)

| عيب `TR_AR_AI` | الإصلاح هنا |
|----------------|-------------|
| جلسات في الذاكرة تضيع عند restart | **كل الحالة في Supabase** عبر المحرّك |
| بلا streaming → فشل >4096 | تحرير تدريجي + تقسيم الرسائل |
| بلا logging | logging مُهيكَل لكل دور/خطأ |
| معالجة أخطاء ضعيفة (تُظهر الخطأ التقني) | رسائل ودّية + retry + fallback، والخطأ التقني في السجل فقط |
| `docker-compose` لا يُعرّف الخدمة | المحرّك جزء من نفس التطبيق — لا تبعية خارجية مفقودة |
| جسر يفوّض لـ Hermes | المحرّك يملك المنطق؛ OpenRouter مباشرة |

## أوامر v1
- بدء/ترحيب · `/progress` (تقدّم نصّي) · `/review` (جلسة SRS) · والباقي **محادثة طبيعية حرة** (الأوامر مكمّلة لا أساسية).

## الأمان
- قصر الوصول على `telegram_user_id` المصرّح (تطبيق شخصي).
- التحقق من توقيع webhook (secret token).

## المرحلة 2 — الصوت (تمهيد)
التيليجرام **ممتاز للرسائل الصوتية أصلاً**: الطالب يرسل صوتاً ← STT ← المحرّك ← رد نصّي + **رسالة صوتية** (TTS) + تقييم نطق. يُبنى فوق نفس العميل دون تغيير المحرّك.
